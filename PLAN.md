# Plan

> Активное направление: генерация Word-документов по шаблонам. Этапы, связанные с ресурсами, поиском и голосовым вводом, отложены.
>
> Статусы этапов и пунктов: `[x]` — подтвержден и завершен, `[ ]` — ожидает подтверждения или находится в работе. Отметки о завершении ставятся только после подтверждения пользователя.
>
> Правило проверки: каждый этап реализации подтверждается соответствующими автотестами. Пункт или этап закрывается только после успешного запуска этих тестов и подтверждения пользователя.

## [x] Этап 1. Согласование идеи
Результат: зафиксированы идея проекта, пользователь, бизнес-логика, стек, граница MVP, риски и вопросы к наставнику.

- [x] Сформулировать идею: web-приложение для генерации Word-документов по шаблонам.
  - Решение: веб-приложение создает типовые Word-документы по одному заранее подготовленному шаблону.
  - Пользователь выбирает данные, заполняет необходимые поля, а приложение сохраняет готовый `.docx` в папке заказчика.
- [x] Определить целевого пользователя и его задачу.
  - Решение: пользователь — сотрудник, который регулярно готовит коммерческие предложения, отчеты и протоколы встреч.
  - Его задача — быстро подготовить и сохранить корректно оформленный `.docx` без ручного копирования и заполнения шаблона в Word.
- [x] Зафиксировать ценность продукта и основную пользовательскую проблему.
  - Решение: приложение сокращает время подготовки типовых документов, снижает число ошибок при ручном заполнении и
  - обеспечивает единый формат файлов.
- [x] Описать основной сценарий: выбор документа, сбор данных, генерация и сохранение `.docx`.
  - Решение: пользователь открывает веб-интерфейс и заполняет форму. Данные заказчика и объекта проекта выбираются из
  - независимых списков базы данных и автоматически подставляются в форму.
  - После проверки обязательных полей результат заполнения фиксируется в базе как запись документа,
  - связывающая выбранные заказчика и объект. Затем приложение формирует и сохраняет `.docx` внутри папки приложения.
- [x] Утвердить границы MVP: что входит в генерацию документов, а что отложено.
  - Решение: MVP включает веб-интерфейс, заполнение обязательных полей одного заранее подготовленного шаблона,
  - валидацию, базу данных, генерацию `.docx` и сохранение файлов в папке приложения.
  - В базе хранятся отдельные данные заказчика и объекта проекта без связи между ними; выбор значений из независимых
  - списков автоматически заполняет форму. Перед генерацией создается запись документа, в которой появляется связь
  - выбранных заказчика и объекта. Документы сохраняются в папке с именем заказчика: если такой папки нет,
  - приложение создает ее; если есть — сохраняет файл в ней. Скачивание и открытие документа для редактирования,
  - голосовой ввод, открытие ресурсов, web-поиск, авторизация, внешние интеграции и автоматизация браузера отложены.
- [x] Утвердить технический стек и обоснование его выбора.
  - Решение: backend реализуется на Python 3.13 с использованием FastAPI. `docxtpl` отвечает за подстановку данных
  - в Word-шаблон, а HTML, CSS и JavaScript без frontend-фреймворка — за форму заполнения. `venv` используется для
  - изолированного окружения. Для хранения данных применяется база данных; конкретная СУБД и схема будут определены
  - на этапе проектирования структуры. MVP должен запускаться на Windows и Linux.
- [x] Зафиксировать технические риски и ограничения.
  - Решение: справочники заказчиков и объектов независимы; их связь создается только в записи документа.
  - Запись документа создается в БД до генерации файла, а при ошибке сохраняется ее статус.
  - Имена папок и файлов должны быть безопасны для Windows и не должны перезаписывать существующие файлы.
  - При отсутствии папки заказчика приложение создает ее. Для совместимости с Windows и Linux пути формируются
  - через `pathlib`, а расположение папки документов задается в конфигурации, без жестко заданных путей в коде.
  - В MVP используется один `.docx`-шаблон с корректными плейсхолдерами; файл не скачивается и не редактируется пользователем.
- [x] Сформулировать вопросы к наставнику, если они остаются.
  - Решение: дополнительные вопросы к наставнику на текущем этапе не требуются.
- [x] Сверить и привести к единому виду `README.md`, `PROJECT_BRIEF.md` и `AGENT_SPEC.md`.
  - Решение: документы синхронизированы с подтвержденным scope MVP: один шаблон, независимые справочники заказчиков и объектов, запись документа в БД до генерации, сохранение `.docx` в папке заказчика и поддержка Windows/Linux. `meet1.md` удален пользователем как неактуальный.

## [x] Этап 2. Проектирование структуры
Результат: определены шаблон документа, обязательные поля, схема базы данных, правила сохранения файлов и структура проекта.

- [x] Утвердить единственный Word-шаблон и список его плейсхолдеров.
  - Решение: используется шаблон `document_templates/CP_ver.01.docx`. Данные заказчика берутся из БД в поля `customer_name` и `director`, данные объекта — в `object_name` и `object_imo`, а `subject` и строки состава работ `proposal_rows` вводятся в форме.
  - Решение: пользователь вводит в форме сумму, срок разработки и срок действия числовыми значениями; они сохраняются в БД. Перед генерацией сервер формирует готовые русскоязычные фразы с правильным склонением для `amount_full_text`, `lead_time_text` и `validity_period_text`.
  - Решение: из шаблона удален блок примечаний. Сумма и сроки подставляются через `amount_full_text`, `lead_time_text` и `validity_period_text`; строки состава работ обрабатываются через `{%tr for row in proposal_rows %}` и `{%tr endfor %}`.
- [x] Определить обязательные поля формы и правила их валидации.
  - Решение: обязательны выбор заказчика с заполненными `customer_name` и `director`, выбор объекта с заполненными `object_name` и `object_imo`, непустой `subject`, хотя бы одна строка состава работ с непустым наименованием, положительная сумма с точностью до двух знаков и положительные целые сроки разработки и действия предложения в днях. Комментарий строки состава работ необязателен. При ошибке валидации запись документа и файл не создаются.
- [x] Выбрать СУБД для MVP и способ работы с ней из Python.
  - Решение: используется SQLite — один файл базы данных внутри приложения. Доступ к БД реализуется через встроенный модуль Python `sqlite3`, без ORM на текущем MVP.
- [x] Спроектировать таблицу заказчиков и ее поля.
  - Решение: таблица `customers` содержит `id`, обязательное уникальное `name`, обязательное `director` и `created_at`. Наименование заказчика подставляется в шаблон и служит основой имени папки для документов.
- [x] Спроектировать таблицу объектов проекта и ее поля.
  - Решение: таблица `project_objects` содержит `id`, обязательное `name`, обязательное уникальное `imo` и `created_at`. Поле `customer_id` отсутствует: заказчик и объект выбираются независимо.
- [x] Спроектировать таблицу документов: она должна хранить данные формы, ссылку на заказчика, ссылку на объект проекта, дату создания и путь к сгенерированному файлу.
  - Решение: таблица `documents` содержит `id`, обязательные `customer_id` и `object_id`, `subject`, `amount_kopecks`, `lead_time_days`, `validity_period_days`, `status` (`pending`, `generated` или `error`), необязательные `file_path` и `error_message`, а также `created_at`. Запись создается до генерации и создает связь выбранных заказчика и объекта.
- [x] Спроектировать таблицу строк состава работ, связанных с документом.
  - Решение: таблица `document_work_items` содержит `id`, обязательный `document_id`, уникальный в пределах документа `position`, обязательный `name` и необязательный `comment`. При удалении документа его строки состава работ удаляются вместе с ним.
- [x] Определить способ начального заполнения справочников заказчиков и объектов.
  - Решение: отдельный идемпотентный Python-скрипт создает БД и добавляет стартовых заказчиков и объекты из заданного в коде списка; повторный запуск не создает дубликаты.
- [x] Определить структуру папок и правило формирования имени `.docx`; документы хранятся в папке заказчика.
  - Решение: корневая папка — `generated_CP`, ее расположение задается конфигурацией. Внутри создается папка с безопасным именем заказчика. Имя файла: `ГГГГ_ММ_ДД_CP_порядковый-номер.docx`, например `2026_07_20_CP_001.docx`. Порядковый номер общий для всех документов; правила отображения списка документов в UI будут определены позднее.
- [x] Согласовать структуру Python-проекта: приложение, модели БД, работа с шаблоном, генерация файлов и web-интерфейс.
  - Решение: код размещается в `app/` с модулями FastAPI, конфигурации, SQLite, моделей, репозиториев и сервисов генерации, русского текста и шаблона. HTML и статические файлы находятся в `app/web/`, Word-шаблон — в `document_templates/`, скрипт инициализации — в `scripts/`, SQLite — в `data/`, сгенерированные файлы — в `generated_CP/`.
