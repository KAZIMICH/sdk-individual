# Идея проекта

## Формулировка
Веб-ассистент с поддержкой голосового ввода, который помогает сотруднику быстрее выполнять рутинные рабочие действия в одном интерфейсе.

Этот проект является одним из модулей более крупной корпоративной системы, которую я разрабатываю вне курса. 
Именно поэтому он выглядит как отдельный ограниченный модуль с понятной минимальной версией и продолжением в основном продукте.

## Пользователь
Сотрудник, который регулярно выполняет повторяющиеся действия в браузере и работает с типовыми документами.

## Ценность
Ассистент сокращает количество ручных действий при выполнении типовых рабочих сценариев.

## Бизнес-логика
Пользователь обращается к ассистенту через единый веб-интерфейс для выполнения типовых рабочих действий. 
Ассистент принимает голосовую или текстовую команду, определяет тип сценария и выполняет разрешенное действие: 
открывает ресурс, выполняет поиск или создает документ по шаблону. 
Для документных сценариев ассистент собирает недостающие данные через уточняющие вопросы и формирует готовый файл.

## Сценарии минимальной версии
- `Открой почту`
- `Найди <запрос>`
- `Создай отчет`
- `Создай КП`
- `Создай протокол встречи`

## Что делает минимальная версия
- открывает заранее заданный ресурс;
- выполняет поиск в сети по запросу;
- создает документ по шаблону через веб-интерфейс;
- принимает голосовую команду;
- принимает текстовую команду как резервный ввод;
- задает уточняющие вопросы, если не хватает данных для документа;
- отдает готовый `.docx` для скачивания;
- хранит историю последних команд.

## Стек и инструменты
- `Python 3.13`
- `venv`
- `SQLite`
- `JSON`
- `Markdown`
- `FastAPI`
- браузерный `Web Speech API` для голосового ввода
- `docxtpl` для генерации документов по шаблонам

## Почему такой стек
Стек остается простым, локальным и проверяемым в рамках курса. 
Он позволяет сделать веб-интерфейс, хранить настройки и шаблоны без отдельной тяжелой инфраструктуры, 
а также закрыть ключевой сценарий генерации `.docx` без доступа к файловой системе пользователя.

## Технологии из курса
- `Python`
- `venv`
- `SQLite`
- `JSON`
- `Markdown`
- `FastAPI`

## Технологии вне курса
- `Web Speech API`
- `docxtpl`

## Зачем нужны технологии вне курса
`Web Speech API` нужен для голосового ввода в браузере. `docxtpl` нужен для подстановки данных в шаблоны `.docx` 
с плейсхолдерами вида `{{...}}` и выдачи готового файла через интерфейс.

## Основной сценарий минимальной версии
Пользователь открывает веб-интерфейс ассистента и голосом или текстом вводит команду `Создай КП`. 
Ассистент распознает намерение, задает несколько уточняющих вопросов, подставляет ответы в шаблон документа и отдает 
готовый `.docx` для скачивания.

## Обязательное
- Web-интерфейс ассистента.
- Кнопка голосового ввода.
- Текстовый ввод как fallback.
- Распознавание 3 типов команд: `open_resource`, `search_web`, `create_document`.
- Настройка списка пользовательских ресурсов.
- 2-3 шаблона документов для генерации.
- Уточняющие вопросы при создании документа, если не хватает данных.
- Генерация готового `.docx` и выдача файла через интерфейс.
- История последних команд.

## Желательно
- Подтверждение перед выполнением команды.
- Озвучивание ответа ассистента.
- Более гибкая настройка ресурсов и шаблонов через UI.
- Расширенный набор документных шаблонов.
- Более точный разбор естественной речи, а не только простых командных шаблонов.

## Вне рамок минимальной версии
- Доступ к локальной файловой системе пользователя.
- Управление desktop-приложениями.
- Работа с содержимым почты.
- Отправка писем.
- Произвольные команды ОС.
- Автоматизация браузера.
- Регистрация, авторизация, роли.
- Интеграции с внешними сервисами кроме обычного открытия URL в браузере.
- Редактирование документов после генерации внутри приложения.

## Техническое ограничение
Для минимальной версии целевой средой демонстрации рассматривается Yandex Browser. Голосовой ввод зависит от поддержки `Web Speech API` в браузере, поэтому не считается полностью совместимым со всеми браузерами.

## Риски и ограничения
- `Web Speech API` может работать нестабильно или зависеть от конкретного браузера и настроек пользователя.
- Голосовой ввод не считается основой надежности проекта, поэтому для всех команд нужен текстовый резервный ввод.
- `docxtpl` требует аккуратной структуры шаблонов `.docx` с плейсхолдерами, иначе рендер документа может ломаться.
- Проект не должен уходить в доступ к локальной файловой системе, автоматизацию рабочего стола или автоматизацию браузера.
- Набор действий должен оставаться ограниченным и проверяемым, без произвольных команд ОС.
- Внешние интеграции кроме браузерного открытия URL не входят в MVP.

## Данные на входе
- Голосовая команда пользователя.
- Текстовая команда пользователя.
- Название ресурса для открытия.
- Поисковый запрос.
- Тип документа.
- Значения для полей шаблона документа.
- Список настроенных ресурсов и шаблонов.

## Данные на выходе
- Открытый web-ресурс.
- Открытая поисковая страница с подставленным запросом.
- Готовый `.docx` по шаблону.
- Текстовый ответ ассистента.
- История выполненных команд.
