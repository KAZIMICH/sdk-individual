# Project Brief

> Текущий этап проекта ограничен генерацией `.docx` по шаблонам. Сценарии открытия ресурсов, поиска и голосового ввода отложены.

## Business Logic
Пользователь обращается к web-интерфейсу для создания документа по единственному шаблону. Он выбирает заказчика и объект проекта из независимых списков БД, заполняет остальные поля, а приложение сохраняет результат.

Перед генерацией приложение создает запись документа, связывающую выбранные заказчика и объект. Затем подставляет значения в шаблон и сохраняет готовый `.docx` в папке заказчика.

Данные хранятся в SQLite: независимые справочники заказчиков и объектов, запись документа и связанные с ней строки состава работ. Сумма и сроки вводятся числовыми значениями, а перед генерацией преобразуются в текст с русскими правилами склонения. Сумма округляется до целого рубля по правилу `0,5` вверх; сумма и сроки выводятся цифрами с разделителем тысяч, затем прописью в скобках и с существительным в нужной форме.

## Пользовательские flow
### Flow 1: создание документа
1. Пользователь выбирает заказчика и объект проекта из независимых списков БД.
2. Приложение автоматически подставляет их данные в форму.
3. Пользователь заполняет оставшиеся обязательные поля.
4. Приложение проверяет форму и создает запись документа в БД.
5. Приложение генерирует `.docx` и сохраняет его в папке заказчика.

## Будущие пользовательские flow

Следующие сценарии зафиксированы как развитие продукта и не входят в текущий MVP генерации документов.

### Flow 2: голосовой помощник
1. Пользователь активирует голосовой ввод и произносит команду.
2. Приложение распознаёт текст команды и определяет поддерживаемый сценарий.
3. При недостатке или неоднозначности данных приложение запрашивает уточнение.
4. Перед выполнением чувствительного действия приложение показывает распознанную команду для подтверждения.

### Flow 3: открытие почтового ящика
1. Пользователь произносит команду открыть почтовый ящик.
2. Приложение открывает заранее настроенный почтовый сервис.
3. Приложение не читает содержимое почты и не отправляет письма без отдельного подтверждённого сценария.

### Flow 4: поиск данных в интернете
1. Пользователь произносит поисковый запрос.
2. Приложение открывает поисковый сервис с распознанным запросом.
3. Пользователь самостоятельно выбирает результат поиска и работает с открытой страницей.

### Flow 5: открытие сайта по голосовой команде
1. Пользователь произносит название сайта или сервиса.
2. Приложение сопоставляет команду с заранее настроенным списком сайтов.
3. Приложение открывает выбранный сайт после подтверждения пользователем.

### Flow 6: создание напоминания голосом
1. Пользователь произносит напоминание с датой и временем.
2. Приложение распознаёт текст, дату и время, затем показывает их для подтверждения.
3. После подтверждения приложение сохраняет напоминание или создаёт рабочую задачу.

## Входные данные
- выбранный заказчик;
- выбранный объект проекта;
- значения для плейсхолдеров шаблона;
- строки состава работ;

## Выходные данные
- запись документа в БД со связью заказчика и объекта;
- готовый `.docx`, сохраненный в папке заказчика.
- список сохранённых записей документов с поиском по заказчику или объекту проекта.

## Ограничения MVP
- один заранее подготовленный шаблон документа;
- только web-формат взаимодействия;
- без доступа к локальной файловой системе;
- без desktop automation;
- без работы с содержимым почты;
- без отправки писем;
- без browser automation.
- без сценариев открытия ресурсов и web-поиска на текущем этапе;
- без голосового ввода на текущем этапе.
- без скачивания и открытия документа для редактирования на текущем этапе.
- запуск на Windows и Linux.
- адаптивный web-интерфейс для desktop, tablet и mobile: форма перестраивается в одну колонку, а список КП — в карточки на узких экранах.
- SQLite-файл и сгенерированные документы хранятся локально и не публикуются в Git.
